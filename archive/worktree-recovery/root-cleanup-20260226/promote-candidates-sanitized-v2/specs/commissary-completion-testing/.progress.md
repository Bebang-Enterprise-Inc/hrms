## Learnings

- Commissary operations **10x simpler than assumed** - only 1 product (FG020 Frozen Milk) needs BOM, not 10+ complex BOMs
- Most commissary products are **outsourced** (Leche Flan, Banana Cinnamon, syrups) or use **pre-packaged ingredients** (jellies, rice crispies)
- FG020 has **2 variants**: Traditional (Nestle Cream + Condensed + Evap) and Griffith (Powder + Water, 10% cheaper, longer prep)
- Only **4 raw materials** to track vs original assumption of 50+ ingredients
- Phase 2 effort reduced from **2-3 days to 2-3 hours** due to simplification
- Frappe BOM best practice: Create via UI (Desk → Manufacturing → BOM), not code
- Stock Entry type matters: "Material Receipt" (Phase 1) vs "Manufacture" (Phase 2 with auto-deduction)
- ERPNext v16 (2026) adds flexible valuation control per industry
- Chrome DevTools MCP already proven for frontend testing (Projects module 100% pass rate)
- Frappe pytest pattern: Test files inherit from `FrappeTestCase`, auto-handles DB transactions
- Phase 3 architecture decision is **critical blocker** - hub-based (2-3 days) vs direct dispatch (0 hours but technical debt)
- Test stock exists in TEST-COMMISSARY - BEI, ready for validation
- **Duplication audit saved 4.5-6.5 days** (72% time savings) by identifying 27 existing APIs, 15 existing DocTypes, 4 existing pages
- Phase 2 has **low duplication risk** (15%) - 3 BUILD + 3 EXTEND tasks vs rebuilding from scratch
- Phase 3 has **medium duplication risk** (40%) if hub-based chosen - should DEFER pending business approval
- Testing has **low duplication risk** (20%) - extend existing E2E baseline, create new backend/integration/performance suites
- Critical gap: Only 3 new APIs needed (`get_bill_of_materials`, `get_production_cost`, `can_we_produce`) vs assumed 6+
- **EXTEND not rebuild**: `submit_production_output()` (add BOM auto-deduction), `get_rm_reorder_alerts()` (add forecasting)
- Architecture decision deferred to Mae/Bryan prevents 2-3 days wasted effort if wrong choice made
- Requirements tagged with **[BUILD]** / **[EXTEND]** / **[DEFER]** based on duplication audit findings
- 15 user stories cover Phase 2 (7), Phase 3 (1 deferred), Testing (7) with testable acceptance criteria
- Success metric: Production cost calculation must match manual calc within ±1% tolerance
- Packaging cost source unresolved: Hardcoded or Item custom field? (Ask Bryan)
- Safety factor for reorder point unresolved: 1.5x or 2x lead time? (Ask Arnold)
- BOM creation via UI enforces **immutability** - submitted BOMs cannot be edited (must duplicate + resubmit to change)
- Stock Entry type "Manufacture" **auto-populates** raw materials from BOM via `se.get_items()` method
- `get_commissary_warehouse()` pattern enables **seamless test-to-prod switching** based on stock availability
- Production cost requires **valuation_rate from Bin** (not Item master) for accurate COGS calculation
- Forecasting needs **7-day consumption window** to smooth out production spikes/dips
- Chrome DevTools MCP supports **visual regression testing** via screenshot comparison (<5% pixel change threshold)
- Performance thresholds: Dashboard <500ms, Production <1s, Inventory <2s (358 items) under 10 concurrent users
- Integration tests validate **end-to-end flow** (production → inventory → fulfillment) vs isolated unit tests
- Frontend component tests use **React Testing Library** for behavior-driven testing (not implementation details)
- Hub-based distribution adds **architectural complexity** (warehouses, trips, routes) vs direct dispatch (technical debt)

## Task Planning Learnings (2026-02-06)

- **POC-first workflow prevents over-engineering** - Phase 1 validates BOM auto-deduction works before writing full test suites
- **Quality checkpoints every 2-3 tasks** catch issues early vs waiting until end
- **Manual BOM creation via Frappe UI** is best practice (not code-generated) - enforces immutability
- **EXTEND not BUILD** saved 45% duplication - `submit_production_output()` and `get_rm_reorder_alerts()` modified, not rebuilt
- **Verification tasks [VERIFY]** use actual commands (`bench run-tests`, `curl`, `/test-full-cycle`) not generic "check quality"
- **Phase 3 architecture decision deferred** to Mae/Bryan prevents 2-3 days wasted effort if wrong choice made
- **Integration tests validate end-to-end flow** (production → inventory → fulfillment) vs isolated unit tests
- **Performance thresholds defined upfront** (<500ms dashboard, <1s production, <2s inventory) prevent regression
- **Visual regression baseline** captured using Chrome DevTools MCP screenshots (<5% pixel change threshold)
- **MANCOM metrics validation** requires manual calculation comparison (±1% tolerance) not just "test passes"
- **Task dependencies explicit** - BOMs must exist before auto-deduction, tests after implementation
- **15 tasks vs 35-40 tasks** (57% reduction) due to duplication audit findings
- **1-2.5 days vs 6-9 days** (72% time savings) from identifying existing 27 APIs, 15 DocTypes, 4 pages
- **Critical gaps identified**: Only 3 new APIs needed (`get_bill_of_materials`, `get_production_cost`, `can_we_produce`) vs assumed 6+
- **Testing strategy multi-layered** - Unit (pytest), Integration (full flows), E2E (Chrome DevTools MCP), Performance (locust), Visual (screenshots)
- **Backwards compatibility enforced** - Items without BOM still use Material Receipt (not Manufacture)
- **Error handling comprehensive** - Insufficient stock, invalid BOM, negative qty all handled gracefully
- **Helper functions extracted** in Phase 2 Refactoring (`get_active_bom()`, `get_bom_raw_materials()`) to follow DRY principle
- **Commit messages reference requirements** - `feat(commissary): ...` links to FR-X, AC-X.X for traceability
- **Test data uses TEST-COMMISSARY** warehouse for safe validation before production
- **All acceptance criteria programmatically verifiable** (AC-1.1 through AC-15.6) via automated checks
