## Learnings

- **Most "missing features" already have backend APIs** -- 80%+ of backend endpoints exist. The gap is frontend integration, not backend logic.
- **Permission pattern is the #1 root cause** -- DocTypes missing Employee role permissions block all staff-level operations.
- **Handover/POS access bugs share a root cause**: Frontend pages call `/api/supervisor/my-stores` which only returns stores for supervisor-role users. Store Staff users get empty response = no store selected = blocked.
- **Kudos has TWO bugs**: (1) Missing write permission on Employee role (can create but not write=save), (2) Category mismatch -- frontend sends "Excellence,Positivity" but DocType expects "Customer Service,Innovation,Going Extra Mile".
- **Leave workflow issues (HR-1/HR-2/HR-3/SUP-2) are Frappe core workflow config**, NOT custom API bugs. Need to verify Leave Application Workflow is configured in Frappe and test employee `reports_to` field is set.
- **6 pages exist but aren't linked in navigation**: Variance Report, Shelf Life, Enrichment Tracker, Store Dashboard exist as pages but missing from sidebar/nav. Just need nav links added.
- **Dropdown conversion follows single pattern**: All 7 text-to-dropdown issues (FQ-1, HR-7, HR-8, COM-2, SUP-4, HR-5) use same Shadcn Combobox + API fetch pattern. Can be batch-implemented.
- **Test data seeding needed for 6 items**: Announcements, Attendance, Payslips, Stock entries, Distribution trips, Approval Queue entries.
- **Deployment sequence matters**: Backend first (backward-compatible permission/field changes), then frontend. Never deploy frontend before backend for new field additions.
- **`custom_area_supervisor` field on Warehouse** controls area dashboard scope. Test warehouses likely don't have this set.
- **Coverage Request API has `resolve_employee()` and `resolve_warehouse()`** already built for fuzzy matching -- frontend just needs to use them via combobox.
- **POS date validation already implemented in backend** (store.py lines 656-714) but frontend never displays the warning response.
- **Cycle count resubmission API already exists** (inventory.py lines 102-156) but frontend doesn't expose resubmit button.
- **Classification: 8 BUG, 22 ENHANCEMENT, 8 NEW FEATURE, 6 DATA/CONFIG** across 44 items.
- **Effort estimate: 3-5 days** for 2 agents working in parallel (backend + frontend).
- **Biggest technical risk**: Leave workflow requires Frappe core Workflow configuration, not just API changes.
- **Denomination feature (C-2) is the most complex new feature** requiring new child DocType + new frontend component.

## Learnings (Requirements Phase)

- **38 user stories distilled from 44 items** -- some items share root causes (H-1+H-2, HR-2+HR-3+SUP-2, AS-1+AS-2) and were grouped into single stories
- **3 items deleted per audit**: denomination child DocType (already exists as fields), store-item mapping DocType (use Item Group), profile new flow (existing Edit Request is bugged, not missing)
- **6 genuinely new builds identified**: deposit_type field, per-fund denominations, voucher amounts, shift templates, ordering RBAC, approval queue creation
- **Dropdown conversion is a batch pattern**: 6 text-to-dropdown items all use identical Shadcn Combobox + API fetch. Should create shared `EntityCombobox` component used by all 6
- **Leave workflow is the biggest config risk**: Frappe Leave Application needs Workflow DocType configuration, not code. If workflow isn't configured, 4 stories (US-20) fail regardless of code quality
- **Deployment sequence is a hard constraint**: Backend DocType field additions MUST deploy before frontend references those fields. Backend changes are backward-compatible so safe to deploy early.
- **5 unresolved questions remain**: shift template names/times, item group mapping, denomination field strategy (individual vs JSON), PR-1 root cause, category source-of-truth direction. All can be resolved during design phase without blocking requirements approval.
- **Per-fund denomination approach**: 30 individual fields (e.g., `pcf_denom_1000`) are more queryable but verbose. 3 JSON fields are cleaner but harder to report on in Frappe. Recommendation: individual fields for consistency with existing `denom_*` pattern.

## Learnings (Design Phase)

- **Navigation links already exist in nav-main.tsx** -- Variance Report (line 310), Shelf Life (line 304), Enrichment (line 338), Store Dashboard are all wired in the sidebar. The UAT issues IV-1, IV-2, SUP-6, SUP-7 are about pages loading data, not missing navigation links. This saved unnecessary nav config work.
- **`_get_area_supervisor_stores()` is the core store scope function** -- used in 10+ places in supervisor.py. The `get_my_stores()` endpoint is just a thin wrapper. Store Staff need a DIFFERENT resolution path (Employee.branch), not a fix to this function.
- **New `get_user_store()` endpoint is the key architectural decision** -- instead of modifying `get_my_stores()` (which is correctly scoped for area supervisors), a new endpoint handles ALL roles cleanly. This avoids regression in the 10+ existing callers.
- **bei-tasks already has 6 Combobox-style selectors** (`branch-selector.tsx`, `user-selector.tsx`, `department-selector.tsx`, `project-selector.tsx`, `tag-selector.tsx`, `task-selector.tsx`). All use identical Shadcn Command+Popover pattern. The new `EntityCombobox` should follow this exact pattern.
- **Denomination: 30 individual fields confirmed as correct approach** -- existing `denom_1000` through `denom_coins` (8 fields) in bei_store_closing_report.json use individual Currency fields. Extending with 30 more (`pcf_denom_*`, `del_denom_*`, `chg_denom_*`) maintains the pattern and allows Frappe Report Builder queries.
- **`submit_order()` does NOT create approval queue entries** -- confirmed by reading store.py lines 152-189. It only creates BEI Store Order with status="Pending Approval" but no BEI Approval Queue record. The `get_unified_approval_queue()` in supervisor.py directly queries BEI Store Order table (not via queue). However, adding explicit queue entries is cleaner for the unified approval flow.
- **BEI Kudos category options:** DocType has `"Teamwork\nCustomer Service\nInnovation\nLeadership\nGoing Extra Mile"`. Frontend must match these exactly.
- **BEI Support Ticket category options:** DocType has `"IT/Technical\nHR Question\nPayroll Issue\nApp Bug\nFeature Request\nOther"`. Frontend must match.
- **BEI POS Upload missing Employee permission confirmed** -- only has System Manager, HR User, Store Staff, Store Supervisor, Area Supervisor roles. No "Employee" role means generic employees can't create.
- **BEI Store Visit Report only has System Manager + HR User** -- missing Area Supervisor and Employee roles entirely.
- **`useEmployee()` hook provides `branch` field** -- this is the key to staff store resolution. `employee.branch` maps to a Warehouse name. No new data needed, just a resolution function.
- **Phase 5 (Leave Workflow) is pure Frappe config, not code** -- Workflow DocType must be configured via bench console or Frappe Desk. No Python or React changes needed.

## Learnings (Tasks Phase)

- **44 items compressed to 14 tasks** by grouping: 3 permission fixes = 1 task, 6 dropdown conversions = 2 tasks, 6 data seeds = 1 task. This avoids micro-task overhead.
- **Store visits page is at `team/visits/page.tsx`**, NOT `supervisor/store-visits/page.tsx` as assumed in design. File path verified via glob.
- **Profile page is at `my-profile/page.tsx`**, NOT `profile/page.tsx`.
- **Enrichment tracker is at `hr/enrichment-tracker/page.tsx`** AND `enrichment/page.tsx` -- two separate pages exist.
- **No `supervisor/store-visits` directory exists** -- store visit dialog is a component (`components/supervisor/new-store-visit-dialog.tsx`) used on `team/visits/page.tsx`.
- **bei-tasks has no TypeScript check command** (`tsc --noEmit` not in package.json scripts). Quality checks limited to `eslint` + `npm run build` (Next.js build catches type errors).
- **Backend quality checks** use `python -m py_compile` for syntax + `json.load()` for JSON validity. No pytest framework for unit tests in this repo (Frappe uses bench test).
- **Critical dependency chain**: Tasks 1.1-1.3 (backend) must deploy (1.5) BEFORE tasks 1.6-1.8 (frontend) can be verified E2E. Frontend can be developed in parallel but only tested after backend deploys.
- **Leave workflow (4.1) is independent** of all other tasks -- can be done any time. Placed in Phase 4 because it's config-only and needs test data from 3.1 to verify.
- **3 highest-leverage new files**: `entity-combobox.tsx` (used 6 times), `use-user-store.ts` (used 5+ times), `denomination-grid.tsx` (used 3 times). These 3 files unlock 14 of the 44 items.
